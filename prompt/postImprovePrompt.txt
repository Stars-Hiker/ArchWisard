## 1. Pacman orphan hook  →  chroot_base.sh

After the "pacman tweaks" section (Color / ParallelDownloads), add a new
section that writes /etc/pacman.d/hooks/orphans.hook:

  section "Pacman hooks"
  mkdir -p /etc/pacman.d/hooks
  cat > /etc/pacman.d/hooks/orphans.hook << 'HOOKEOF'
  [Trigger]
  Operation = Install
  Operation = Upgrade
  Operation = Remove
  Type = Package
  Target = *

  [Action]
  Description = Checking for orphaned packages...
  When = PostTransaction
  Exec = /bin/bash -c 'o=$(pacman -Qdtq 2>/dev/null); [ -n "$o" ] && echo "WARNING: orphaned packages: $o" || true'
  HOOKEOF
  ok "pacman orphan hook installed"

This is idempotent (cat > overwrites), safe, and adds zero dependencies.

## 2. paru configuration file  →  chroot_base.sh

In the "AUR helper" section, after the AUR helper is installed via makepkg,
write a default ~/.config/paru/paru.conf (as the new user, not root).

Only write it if AUR_HELPER starts with "paru" (paru or paru-bin).
Only write it if the file does not already exist.

Add this block immediately after the `ok "${AUR_HELPER} installed"` line:

  if [[ "$AUR_HELPER" == paru* ]]; then
      local _paru_conf="/home/${USERNAME}/.config/paru/paru.conf"
      if [[ ! -f "$_paru_conf" ]]; then
          mkdir -p "/home/${USERNAME}/.config/paru"
          cat > "$_paru_conf" << 'PARUEOF'
  [options]
  BottomUp
  SudoLoop
  CombinedUpgrade
  CleanAfter
  PARUEOF
          chown -R "${USERNAME}:${USERNAME}" "/home/${USERNAME}/.config/paru"
          ok "paru.conf written"
      fi
  fi

Risk: none — pure file write, conditional on paru being the chosen helper.

## 3. Power management step  →  identity.sh + chroot_gen.sh + chroot_base.sh

### 3a. New question in choose_extras (identity.sh)

Add a new subsection in choose_extras(), after the "Peripherals" subsection

  blank
  _ui_subsection "Power management"

  pm_sel=$(choose_one \
      "TLP      — laptop: battery optimisation (recommended for laptops)" \
      "TLP      — laptop: battery optimisation (recommended for laptops)" \
      "tuned    — desktop: CPU governor profiles" \
      "none     — no power manager" \

      *)     POWER_MGMT="none"  ;;
  esac
  ok "Power management: ${POWER_MGMT}"

Default should pre-select TLP when GPU_VENDOR gives no desktop signal, or
leave it to the user — do not auto-detect; just present the choice.

### 3b. Add POWER_MGMT to globals (archwizard.sh)

### 3c. Serialize POWER_MGMT (chroot_gen.sh)

In _serialize_chroot_vars(), add to the heredoc under "Desktop & extras":

### 3d. Install and enable in chroot (chroot_base.sh)

Add a new section after the "CUPS printing" section:

  if [[ "$POWER_MGMT" == "tlp" ]]; then
  section "Power management — TLP"
  # Mask tuned before enabling TLP — they fight over the CPU governor.
  systemctl stop    tuned.service 2>/dev/null || true
  systemctl disable tuned.service 2>/dev/null || true
  systemctl mask    tuned.service 2>/dev/null || true
  pacman -S --noconfirm --ask 4 --needed tlp tlp-rdw
  # Mask rfkill units — TLP manages radio kill switches directly.
  systemctl mask systemd-rfkill.service systemd-rfkill.socket
  systemctl enable tlp.service
  ok "TLP enabled"

  elif [[ "$POWER_MGMT" == "tuned" ]]; then
  section "Power management — tuned"
  pacman -S --noconfirm --ask 4 --needed tuned
  systemctl enable tuned.service
  ok "tuned enabled (profile: balanced — set after first boot with: tuned-adm profile balanced)"
  fi

Risk: TLP/tuned masking is irreversible within the session but that is the
correct and documented behaviour. The user already chose the power manager.

## 4. SSH hardening drop-in  →  chroot_base.sh

Add a new section after "Core services", before "Snapper":

  section "SSH hardening"
  mkdir -p /etc/ssh/sshd_config.d
  cat > /etc/ssh/sshd_config.d/99-hardening.conf << 'SSHEOF'
  # Generated by ArchWizard — edit after first boot.
  PermitRootLogin no
  X11Forwarding no
  MaxAuthTries 3
  # PasswordAuthentication is left ON intentionally.
  # Set to "no" only after confirming key-based auth works.
  PasswordAuthentication yes
  SSHEOF
  ok "SSH hardening drop-in written (sshd not enabled — activate with: systemctl enable sshd)"

This is unconditional — it writes a config file that only takes effect if
the user later enables sshd. It is safe and idempotent.

### 6d. Install and configure in chroot (chroot_base.sh)

Add a new section after "User accounts":

  if [[ "$USER_SHELL" == "zsh" ]]; then
  section "ZSH shell"
  pacman -S --noconfirm --ask 4 --needed zsh

  # Add zsh to /etc/shells if not already listed
  grep -qxF "$(command -v zsh)" /etc/shells \
      || echo "$(command -v zsh)" >> /etc/shells

  # Set zsh as the login shell for the new user
  chsh -s "$(command -v zsh)" "${USERNAME}"
  ok "ZSH set as default shell for ${USERNAME}"

  # Write a minimal .zshrc — does not block dotfiles from overriding it
  local _zshrc="/home/${USERNAME}/.zshrc"
  if [[ ! -f "$_zshrc" ]]; then
      cat > "$_zshrc" << 'ZSHEOF'
  # ArchWizard baseline .zshrc — safe to override via dotfiles.

  HISTFILE=~/.zsh_history
  HISTSIZE=10000
  SAVEHIST=10000
  setopt HIST_IGNORE_DUPS HIST_IGNORE_SPACE SHARE_HISTORY

  # Basic completion
  autoload -Uz compinit && compinit

  # Prompt (replace with starship/powerlevel10k via dotfiles)
  PS1='%F{141}%n@%m%f %F{51}%~%f %# '
  ZSHEOF
      chown "${USERNAME}:${USERNAME}" "$_zshrc"
      ok "Minimal .zshrc written for ${USERNAME}"
  fi
  fi

Risk: `chsh` inside chroot works correctly. The .zshrc guard (`[[ ! -f ]]`)
ensures dotfiles deployed in Phase 14 are not clobbered.

      local stow_url
      stow_url=$(input_gum \
          "Git repository URL  (e.g. https://github.com/user/dotfiles.git)" \
          "")
      if [[ -z "$stow_url" || "$stow_url" == "$BACK" ]]; then
          info "Dotfiles skipped."
          return 0

          run "arch-chroot /mnt sudo -u ${USERNAME} bash ${_inst}"
          ok "Stow dotfiles deployed via install.sh"

          warn "After reboot: cd ~/.dotfiles && stow <modules>"
      fi
      ;;

Also add the offline USB fallback for ALL git-based methods (bare repo, stow,
chezmoi). Before attempting any git clone, check connectivity:

  if ! curl -s --max-time 5 https://github.com > /dev/null 2>&1; then
      warn "GitHub unreachable — searching for offline copy on USB..."
      local _usb_src
      _usb_src=$(find /run/media /mnt -maxdepth 3 \
                 -name "dotfiles*" -type d 2>/dev/null | head -1)
      if [[ -n "$_usb_src" ]]; then
          _dl_from_local "$_usb_src" "$home_dir" "$USERNAME"
          return 0
      fi
      warn "No offline dotfiles found. Skipping."
      return 0
  fi

Place this check at the top of deploy_dotfiles(), before the method_sel
choose_one, so it applies uniformly.

Risk: the USB search is read-only. The install.sh execution runs as the
target user inside arch-chroot — same pattern as the AUR helper step.

## 8. PipeWire user service enabling  →  chroot_base.sh

The current PipeWire section installs packages and does nothing else.
PipeWire user services (pipewire.service, wireplumber.service) are
socket-activated per-user and do NOT need systemctl enable in most setups —
they are activated by the socket unit on first use.

However, explicitly enabling them for the new user ensures they start on
first login even without a display manager socket trigger.

After the existing `ok "PipeWire installed"` line, add:

  # Enable PipeWire user services for the new user.
  # Must run as the target user, not root. machinectl not available in chroot,
  # so we write a systemd user preset instead.
  mkdir -p "/home/${USERNAME}/.config/systemd/user/default.target.wants"
  for _svc in pipewire.service pipewire-pulse.service wireplumber.service; do
      local _src="/usr/lib/systemd/user/${_svc}"
      local _dst="/home/${USERNAME}/.config/systemd/user/default.target.wants/${_svc}"
      if [[ -f "$_src" && ! -e "$_dst" ]]; then
          ln -sf "$_src" "$_dst"
          ok "PipeWire user service linked: ${_svc}"
      fi
  done
  chown -R "${USERNAME}:${USERNAME}" "/home/${USERNAME}/.config"

Risk: This is the standard mechanism for enabling user services without a
running session. It is idempotent (ln -sf, -e guard). It only applies when
USE_PIPEWIRE=true (already guarded by the surrounding if block).

## Explicitly excluded (personal / context-specific)

- QEMU/KVM stack — optional, niche; add only if a future "Virtualisation" step
  is created with its own YES/NO gate.
- Neovim init.lua — too personal for an installer.
- Specific UFW rules (Deluge ports, LAN CIDR) — user-specific; the installer
  only needs to enable the firewall, not configure app rules.
- hyprexpo via hyprpm — requires a running Hyprland session; cannot run from chroot.
- mirrorlist backup before reflector — low value; reflector --save overwrites
  atomically and pacman caches the previous state.
